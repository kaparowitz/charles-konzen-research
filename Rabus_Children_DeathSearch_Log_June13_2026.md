## Method
Each Gotha burial volume on Archion has a **Namensregister (alphabetical name index)** at the END
(columns: given name/relation · Jahr · Nro · Seite · Surname). Open volume → jump to the index →
find the **R block** (it sits just before the S block) → scan for "Rabus". Viewer tips: image-number
input is stale-canvas-prone — after each jump click the "+" zoom once to force a redraw before reading.

### Memory-image upload — re-investigated 15 June 2026 (why the old method no longer works)
Jed correctly noted memories WERE uploaded in earlier sessions. Re-checked exactly how, and what
changed. **The Archion→canvas half now works; the FamilySearch half is the new blocker.**

**How it actually worked before (per the KHB2-5J5 vital-records notes):** the memory that succeeded
(the 1833 marriage page) was **not** uploaded by pure JS injection — that attempt pulled the *wrong*
page and was abandoned. It succeeded because **Jed downloaded the full-resolution page from Archion's
in-app Download button to a file, and the Claude-in-Chrome `file_upload` tool pushed that file into an
`<input type=file>` on FamilySearch.** `file_upload` *requires* a DOM file-input ref to target.

**What I proved works now:** compositing the Archion page in-browser. On the Archion viewer tab I drew
the page image to a `<canvas>` → `toDataURL('image/jpeg')` (799×625, untainted — same-origin) and
stashed it in `window.name`; it survived the cross-origin navigation to FamilySearch intact (227 KB
data URL present on the FS page). So the image-capture half of the pipeline is functional.

**What's different / the new blocker — FamilySearch changed its uploader.** The current FS build
exposes **no programmatically-reachable upload target**, verified on three surfaces:
- Person Memories tab (KH62-V94): `input[type=file]` count = **0** before *and* after opening
  "Add Memories → Upload." Patching `HTMLInputElement.prototype.click` caught **0** calls → FS no
  longer clicks a hidden input; "Upload" invokes the browser's **native File System Access API**
  (`showOpenFilePicker`), which by security design cannot be driven by script (no synthetic file
  selection, the OS picker result can't be fabricated).
- Dispatching synthetic `dragenter`/`dragover` (with a `DataTransfer` carrying a file) on the
  "Do you have Memories of this person?" area produced **no drop overlay and no handler reaction** →
  no JS drop listener to feed either.
- `/photos/upload` → 404; `/memories/gallery` → also **0** file inputs.
Because there is neither an `<input>` (so `file_upload` and `change`-injection are dead) nor a JS drop
handler (so a synthetic `drop` is dead), the in-browser pipeline has no FamilySearch-side endpoint on
this build. This is a FamilySearch platform change since the earlier session, not a tooling gap I can
code around.
